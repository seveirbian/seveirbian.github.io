最近科研任务紧张，我们就阅读一下和我的科研方向相关的文章。介绍一篇FAST’16的文章，文章针对Docker容器第一次启动慢的问题，提出了按需获取的方法。

文章来源

Slacker: Fast Distribution with Lazy Docker Containers (FAST’16)

简介

容器作为一种轻量级虚拟化方法,为用户提供了轻量级的隔离环境。容器本质上是享受了虚拟化资源的进程，因此容器的启动应该和进程一样快才对。但是当前容器的部署十分缓慢，原因在于容器文件系统的部署很慢。作者首先对Dockerhub上排名前57的容器镜像的部署过程做了详细的分析，发现1）容器镜像的下载占据了容器启动时间的76%；2）但是容器提供服务所需要的数据只有6.4%；3）不同镜像间的块去重比对每个镜像做gzip压缩能够获得更佳的压缩率。作者根据这三点发现，实现了Docker存储驱动Slacker。Slacker支持按需获取数据，块级别去重，缓存共享。实验结果显示，Slacker能够提升部署时间为5倍，提供开发部署时间为20倍。

背景

Docker作为容器管理工具，已经越来越火爆。

问题

作为轻量级虚拟化方案，Docker容器的第一次启动速度相比进程太慢了。

解决方案

设计HelloBench测试工具，测试并分析Docker容器第一次启动时的流程

HelloBench测试容器启动时间的方法：1）容器执行最简单的任务（编程语言容器执行编译helloworld的任务、发行版镜像执行echo hello）；2）容器返回准备就绪信息（数据库和web服务器任务返回就绪信息）。

在测试之后，得到的分析结果如下：

  1. 镜像数据：使用blktrace工具测试启动过程中容器实际读取的数据。镜像在未压缩时平均体积为329MB、压缩后为117、实际读取的数据为20MB。启示：在容器执行过程中实际读取的数据量无论是和压缩或者未压缩的数据比都很小。镜像数据在网络中传输是以压缩的格式，在写磁盘的时候是解压格式，这对于网络和磁盘来说负载都很大。因此一种减少开销的方法就是按需获取数据。

  2. Pull、Push、Run操作的性能：作者分别测试了这三种操作的性能，发现Run最快（0.97s）、Pull（16s）和Push（61s）很慢。启示：pull操作决定了容器的启动时间（pull占据了76%的启动时间）。使用push操作来发布镜像的开销太大，特别是那些迭代开发应用的开发人员。大部分的push操作存储驱动的Diff功能实现的，大部分的pull操作是由存储驱动的ApplyDiff功能实现的。优化这两个驱动功能能够显著提升性能。

  3. 镜像层：一个镜像都是由多个层构成的。作者测试了不同层对数据访问的性能的影响，发现层数越深文件打开速度越慢；写文件时，底层文件越大写速度越慢；当前容器镜像的数据通常都是放在层数较深的位置。启示：在层次文件系统中，深层的数据访问较慢，同事容器镜像通常在深层存储数据。因此将镜像压缩成一层能够显著减少性能损失。但是单层的镜像失去了层次文件系统带来的优势。

  4. 缓存：考虑同一台主机上的同一个容器可能会运行多次。作者测试发现读/写比例为88/12；对读来说，99%的数据可以从缓存受益。启示：同一镜像启动多次过程中，读取的数据是一样的。在大集群中，重复的运行是不常见的，除非容器的放置受到严格限制（负载均衡、错误隔离);但是对工具容器（pyshon、gcc）或者小型集群来说，重复的启动是更常见的。后一种情形可以从缓存中受益。

Slacker设计

  1. 目标：1）push和pull加快；2）不对长时间运行的容器引入额外开销；3）尽量复用当前的文件系统；4）充分利用现代存储服务器的元语；5）只对存储驱动做修改。

  2. 存储：为了避免下载无用的数据，Slacker将所有的容器数据存储在NFS服务器（VMStore）中，NFS服务器被所有节点共享。启动容器时，节点可以按需获取容器数据。每个容器在NFS服务器中表示为一个NFS文件。这个NFS可以使用Linux loopback将其挂载为虚拟块设备，为容器提供根文件系统。

  3. VMStore集成：作者使用VMStore作为NFS。VMStore提供了restful的api，并且拥有强大的COW功能：snapshot和clone。snapshot为一个NFS文件创建一个只读的快照，clone从一个快照创建一个NFS文件。Slacker利用这两个元语，分别实现了Diff和ApplyDiff功能。

  4. 对Snapshot和Clone的进一步优化：容器镜像是由多层构成的，因此Docker拉取一个镜像时，它会获取所有的层，并将其转交给驱动的ApplyDiff功能。但是Slacker是一个扁平化的镜像格式，因此它只需要最上层的镜像就足够了。因此Slacker在执行ApplyDiff功能时不直接使用clone，而是在调用Get功能时才实际执行Clone操作。为了防止多个镜像基于同一个镜像的情况，Slacker只在第一次时执行snapshot，并在后续的情况下复用该快照。这样就能够显著减少snapshot和clone的操作数量。

  5. Linux内核的修改：从同一个镜像启动的多个容器会读取相同的数据，因此缓存共享是十分有用的。由于Slacker是在VMStore内部实现的COW（在本地文件系统之下），因此缓存状态无法被客户机感知。为了让客户机能够感知，作者修改了客户机Linux的page cache。

测试

  1. HelloBench：提升部署时间5.3倍，提升开发周期20倍。

  2. 长时间运行的性能：Slacker与AUFS驱动性能相差无几。

  3. 缓存：Slacker比AUFS性能稍差。AUFS：Slacker：Slacker+内核改进=0.67：1.71：0.97。

  4. 扩展性：扩展性不再与镜像体积相关，而是与客户机数目相关

我的总结

Lazy fetch确实可以显著提升容器的启动时间，但是后续文件的获取可能会遭受较大的延迟。同时Slacker在面对多次启动的情形时，也会显得有点捉襟见肘。
